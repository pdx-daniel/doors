# Web build & deploy

This guide covers everything you need to know about building the doors web app with Webpack, developing locally with hot reload, and deploying the production bundle to a static host.

---

## 1. Overview

The doors web app shares its codebase with iOS and Android via `react-native-web`. Webpack is the bundler: it compiles React Native components, resolves platform-specific files (`.web.tsx`), and produces a static bundle that runs in any browser.

**Build pipeline at a glance:**

```
index.web.js  ──>  Babel (envName: 'web')  ──>  Webpack  ──>  web-dist/
                                                              ├── bundle.js
                                                              ├── index.html
                                                              ├── basemaps/
                                                              └── (image assets)
```

Key packages:

| Package | Purpose |
|---------|---------|
| `webpack` ^5.98.0 | Bundler |
| `webpack-cli` ^6.0.1 | CLI for production builds |
| `webpack-dev-server` ^5.2.0 | Dev server with HMR |
| `babel-loader` ^9.2.1 | Transpile TS/JSX for web |
| `html-webpack-plugin` ^5.6.3 | Generates `index.html` |
| `copy-webpack-plugin` ^14.0.0 | Copies basemap tiles |
| `css-loader`, `postcss-loader`, `style-loader` | CSS pipeline |
| `react-native-web` ^0.21.2 | React Native primitives for the browser |
| `react-dom` 19.1.0 | DOM renderer (peer of react-native-web) |

## 2. Development server

Start the Webpack dev server with:

```bash
bun run dev:web
```

This runs `webpack serve --config webpack.config.js --mode development` inside `apps/mobile`. The server starts on **port 3001** and opens `http://localhost:3001` in your browser.

### Hot module replacement

The dev server enables `hot: true`, so changes to source files appear in the browser without a full page reload. CSS changes apply instantly; component updates trigger a re-render while preserving application state.

### Static files

The `public/` directory is served as the dev server's static root:

```js
// webpack.config.js
devServer: {
  static: { directory: path.join(appRoot, 'public') },
}
```

This means files like `public/basemaps/basemap.pmtiles` are available at `/basemaps/basemap.pmtiles` during development without needing the CopyWebpackPlugin.

### SPA fallback in dev

`historyApiFallback: true` is enabled. When you refresh at `/settings` or `/dummy-1`, the dev server serves `index.html` instead of returning a 404. This is necessary because React Navigation handles routing client-side.

### Before you start

Make sure the API server is running:

```bash
bun run dev:server
```

The web client fetches data from `http://localhost:3000` by default (configurable via `DOORS_API_URL` — see section 6).

## 3. Production build

Generate a production-optimized bundle:

```bash
bun run build:web
```

This runs `webpack --config webpack.config.js --mode production` inside `apps/mobile`. Output goes to `apps/mobile/web-dist/`.

### What the production build does differently

- **Minification:** Webpack 5 uses `TerserPlugin` by default in production mode to minify `bundle.js`.
- **Tree shaking:** Dead code elimination via ES module static analysis.
- **No dev server:** No HMR, no dev middleware.
- **Environment variables baked in:** `DOORS_API_URL` is set at build time via `DefinePlugin`.

### Output structure

```
apps/mobile/web-dist/
  ├── index.html         (generated by HtmlWebpackPlugin)
  ├── bundle.js          (compressed application bundle)
  ├── basemaps/
  │   └── basemap.pmtiles  (copied via CopyWebpackPlugin)
  └── *.png / *.svg      (image assets co-located with bundle)
```

The `web-dist/` directory is gitignored. It is the deployment artifact.

## 4. Webpack config deep-dive

The config lives at `apps/mobile/webpack.config.js`.

### Entry

```js
entry: path.join(appRoot, 'index.web.js')
```

`index.web.js` is the web-specific entry point. It registers the root React Native component and mounts it into the DOM:

```js
import {AppRegistry} from 'react-native'
import App from './App'
import {name as appName} from './app.json'

AppRegistry.registerComponent(appName, () => App)
AppRegistry.runApplication(appName, {
  initialProps: {},
  rootTag: document.getElementById('root'),
})
```

### Output

```js
output: {
  path: path.join(appRoot, 'web-dist'),
  filename: 'bundle.js',
}
```

All output lands in `web-dist/`. The filename is a single `bundle.js` — code splitting is not currently configured.

### Resolve

```js
resolve: {
  mainFields: ['browser', 'module', 'main'],
  modules: [
    path.join(appRoot, 'node_modules'),
    path.join(monorepoRoot, 'node_modules'),
    'node_modules',
  ],
  alias: {
    'react-native$': 'react-native-web',
    '@': path.join(appRoot, 'src'),
    '@doors/api': path.join(monorepoRoot, 'packages/api/src/index.ts'),
    '@doors/server': path.join(monorepoRoot, 'apps/server/src/app.ts'),
    [path.join(appRoot, 'src/components/MapView')]: path.join(
      appRoot, 'src/components/MapView.web.tsx',
    ),
  },
  extensions: ['.web.tsx', '.web.ts', '.web.js', '.tsx', '.ts', '.jsx', '.js', '.json'],
}
```

Key details:

- **`mainFields: ['browser', 'module', 'main']`** — Webpack prioritizes browser-friendly module entries. This is essential for libraries like `react-native-web` that ship a `browser` field.
- **`alias`** — `react-native` is redirected to `react-native-web`. The `@` alias maps to `src/` for clean imports. Monorepo packages (`@doors/api`, `@doors/server`) are referenced by their source files so Babel can transpile them alongside app code. The `MapView` alias forces the web platform implementation (see section 5).
- **`extensions`** — `.web.*` files have the highest priority, ensuring platform-specific code is selected before the shared fallback.

### Module rules

```js
module: {
  rules: [
    { test: /\.m?js$/, resolve: { fullySpecified: false } },
    { test: /\.(js|jsx|ts|tsx)$/, include: [...], use: { loader: 'babel-loader', options: { envName: 'web' } } },
    { test: /\.(png|jpe?g|gif|svg)$/i, type: 'asset/resource' },
    { test: /\.css$/, use: ['style-loader', 'css-loader', 'postcss-loader'] },
  ],
}
```

| Rule | Purpose |
|------|---------|
| `fullySpecified: false` | Allows extensionless ESM imports from packages like React Navigation (webpack 5 requires explicit extensions by default for ESM). |
| `babel-loader` | Transpiles TS/JSX using the `'web'` env preset. The `include` array covers `appRoot`, `packages/api`, and certain `node_modules` patterns (`@doors`, `nativewind`, `@gluestack-ui`). |
| `asset/resource` | Emits image files to the output directory and replaces references with their public URL. |
| CSS pipeline | `style-loader` injects CSS into the DOM via `<style>` tags. `css-loader` resolves `@import` and `url()`. `postcss-loader` runs Tailwind CSS v4 and autoprefixer. |

### Plugins

```js
plugins: [
  new HtmlWebpackPlugin({ template: path.join(appRoot, 'public/index.html') }),
  new webpack.DefinePlugin({
    'process.env.DOORS_API_URL': JSON.stringify(process.env.DOORS_API_URL ?? 'http://localhost:3000'),
  }),
  new CopyWebpackPlugin({
    patterns: [{
      from: path.join(appRoot, 'public/basemaps'),
      to: 'basemaps',
      noErrorOnMissing: true,
    }],
  }),
]
```

| Plugin | Role |
|--------|------|
| **HtmlWebpackPlugin** | Generates `web-dist/index.html` from the `public/index.html` template and injects a `<script>` tag for `bundle.js`. |
| **DefinePlugin** | Replaces `process.env.DOORS_API_URL` in source code with the literal string value. Dead code elimination removes the fallback branch when the env var is set. |
| **CopyWebpackPlugin** | Copies `public/basemaps/basemap.pmtiles` (and any sibling files) into `web-dist/basemaps/`. The `noErrorOnMissing: true` flag prevents build failures when the directory does not exist (e.g., before running `basemap:refresh`). |

## 5. Platform-specific code

The project uses `.web.tsx` / `.native.tsx` file extensions to separate platform implementations. Webpack's `resolve.extensions` array prioritizes `.web.*` files:

```js
extensions: ['.web.tsx', '.web.ts', '.web.js', '.tsx', '.ts', '.jsx', '.js', '.json']
```

For example, `MapView` has two implementations:

- `src/components/MapView.native.tsx` — uses `@maplibre/maplibre-react-native` (native MapLibre)
- `src/components/MapView.web.tsx` — uses `maplibre-gl` (browser MapLibre GL JS)

The resolve alias in the Webpack config forces the web implementation:

```js
alias: {
  [path.join(appRoot, 'src/components/MapView')]: path.join(appRoot, 'src/components/MapView.web.tsx'),
}
```

This means any import of `@/components/MapView` resolves to `MapView.web.tsx` in the web bundle, regardless of the import source pattern. The Metro bundler (for iOS/Android) uses the opposite alias via `react-native.config.js` to select `MapView.native.tsx`.

### Adding a new platform-specific component

1. Create two files: `MyComponent.web.tsx` and `MyComponent.native.tsx`.
2. In the Webpack config, add an alias from the base path to the `.web.tsx` file.
3. In `babel.config.js`, there is no additional config needed for web — the resolve alias does all the work.
4. Import as `@/components/MyComponent` (without extension) in shared code.

## 6. Environment variables

### `DOORS_API_URL`

Controls the API origin that the web client connects to.

| Context | Default value | How to override |
|---------|---------------|-----------------|
| Local dev | `http://localhost:3000` | Set `DOORS_API_URL` in `.env` or in the shell before running `bun run dev:web` |
| Production build | `http://localhost:3000` | Set `DOORS_API_URL` in the build environment (CI, Docker, etc.) |

This variable is baked into the bundle at **build time** via `DefinePlugin`:

```js
new webpack.DefinePlugin({
  'process.env.DOORS_API_URL': JSON.stringify(
    process.env.DOORS_API_URL ?? 'http://localhost:3000',
  ),
})
```

Because the replacement is a literal string substitution, the fallback (`DEFAULT_API_URL` from `packages/api/src/constants.ts`) is never reached in production — the value is determined when `webpack` or `webpack serve` runs.

### Setting it for production

```bash
DOORS_API_URL=https://api.doors.example.com bun run build:web
```

Or in a CI pipeline:

```yaml
- run: bun run build:web
  env:
    DOORS_API_URL: https://api.doors.example.com
```

### Other environment variables

Webpack does not read `.env` files automatically. If you need additional env vars, either:

- Add them to `DefinePlugin` in `webpack.config.js`
- Use a separate mechanism (e.g., runtime config fetched from the API)

## 7. Static hosting

The `web-dist/` directory is a fully static website. It can be deployed to any static hosting provider.

### What to deploy

Deploy the entire contents of `apps/mobile/web-dist/`:

```
web-dist/
  index.html
  bundle.js
  basemaps/basemap.pmtiles
  (<image assets as needed>)
```

### Deploy examples

#### Netlify

Create a `netlify.toml` in the monorepo root (or in `apps/mobile/`):

```toml
[build]
  command = "cd apps/mobile && bun run build:web"
  publish = "apps/mobile/web-dist"

[[redirects]]
  from = "/*"
  to = "/index.html"
  status = 200
```

The redirect rule enables SPA fallback (see section 8).

#### Vercel

Add a `vercel.json` in the monorepo root:

```json
{
  "buildCommand": "cd apps/mobile && bun run build:web",
  "outputDirectory": "apps/mobile/web-dist",
  "rewrites": [
    { "source": "/(.*)", "destination": "/index.html" }
  ]
}
```

#### AWS S3 + CloudFront

1. Sync `web-dist/` to an S3 bucket:

```bash
aws s3 sync apps/mobile/web-dist/ s3://your-bucket-name/ --delete
```

2. Configure S3 static website hosting or CloudFront.
3. Set the error document to `index.html` for SPA fallback:

```bash
aws s3 website s3://your-bucket-name/ \
  --index-document index.html \
  --error-document index.html
```

#### Nginx / Apache on a VPS

**Nginx:**

```nginx
server {
  listen 80;
  server_name app.example.com;
  root /var/www/doors/web-dist;

  index index.html;

  location / {
    try_files $uri $uri/ /index.html;
  }
}
```

**Apache:**

```apache
<IfModule mod_rewrite.c>
  RewriteEngine On
  RewriteBase /
  RewriteRule ^index\.html$ - [L]
  RewriteCond %{REQUEST_FILENAME} !-f
  RewriteCond %{REQUEST_FILENAME} !-d
  RewriteRule . /index.html [L]
</IfModule>
```

## 8. SPA routing

The doors web app uses React Navigation for client-side routing. Tabs are mapped to URL paths in `src/navigation/linking.ts`:

```ts
export const linking: LinkingOptions<RootTabParamList> = {
  prefixes: [getWebPrefix()],
  config: {
    screens: {
      Map: '',
      Dummy1: 'dummy-1',
      Settings: 'settings',
    },
  },
}
```

This means:

| Path | Screen |
|------|--------|
| `/` | Map tab |
| `/dummy-1` | Dummy1 tab |
| `/settings` | Settings tab |

Because navigation happens entirely in the browser (no server-side rendering), the server **must** serve `index.html` for all routes. If a user navigates to `/settings` and refreshes, the server must not return 404.

### Dev server

Webpack dev server has `historyApiFallback: true` set, which handles this automatically.

### Production host

Configure your static host to fallback to `index.html`:

| Provider | Configuration |
|----------|---------------|
| **Netlify** | `[[redirects]] from = "/*" to = "/index.html" status = 200` |
| **Vercel** | `{ "rewrites": [{ "source": "/(.*)", "destination": "/index.html" }] }` |
| **S3** | Set error document to `index.html` |
| **CloudFront** | Custom error response: 403 → `/index.html` (200) |
| **Nginx** | `try_files $uri $uri/ /index.html;` |
| **Apache** | `RewriteRule . /index.html [L]` |

### What happens without SPA fallback

A user who refreshes at `/settings` gets a **404 page** from the host, because no physical file exists at `web-dist/settings/index.html`. The app breaks. Always set up the fallback rule.

## 9. Basemap tiles in production

The map component requires a PMTiles basemap file for rendering. During local development, the file is served from `public/basemaps/basemap.pmtiles` by the dev server's static directory.

In production builds, the **CopyWebpackPlugin** copies the basemap into the output:

```js
new CopyWebpackPlugin({
  patterns: [{
    from: path.join(appRoot, 'public/basemaps'),
    to: 'basemaps',
    noErrorOnMissing: true,
  }],
})
```

The result: `web-dist/basemaps/basemap.pmtiles` is available at `/basemaps/basemap.pmtiles` in the deployed app.

### Generate the basemap

```bash
bun run basemap:refresh
```

This runs Planetiler via Docker and produces the file at `public/basemaps/basemap.pmtiles`. See [Basemap tiles guide](./04-basemap-tiles.md) for details.

### If the basemap is missing

- The build will not fail (thanks to `noErrorOnMissing: true`).
- The map component will show a blank background or an error state.
- Check that `bun run basemap:refresh` was run before the build, or verify the file exists at `apps/mobile/public/basemaps/basemap.pmtiles`.

### Large file considerations

PMTiles files can be hundreds of megabytes. For production:

- **CDN offloading:** Host the PMTiles file on a CDN or S3 and configure the map component to fetch it from the CDN URL. This reduces your origin server load and speeds up tile delivery.
- **Compression:** PMTiles is already highly compressed. Do not apply additional gzip at the web server level — map libraries request specific byte ranges, and gzip breaks range requests.
- **Cache headers:** Set a long `Cache-Control` header (e.g., `max-age=31536000, immutable`) on the PMTiles file since it changes infrequently.

### Alternative: serve tiles externally

If you prefer not to bundle the basemap, update the map component to use an online tile source (e.g., MapTiler, Mapbox, or an S3-hosted PMTiles URL) instead of a local path.

## 10. Troubleshooting

### Build fails with "Module not found"

**Check the alias paths** in `webpack.config.js`. The `@doors/api` and `@doors/server` aliases point to monorepo source files. If the monorepo layout changes, update the paths accordingly.

**Clear Webpack cache:**

```bash
rm -rf apps/mobile/node_modules/.cache
bun run build:web
```

### Bundle is too large

The production bundle is a single `bundle.js` file. If it grows large:

1. **Check what is bundled:** Run `webpack --mode production --profile --json > stats.json` and analyze with `webpack-bundle-analyzer`.
2. **Code splitting:** Consider adding `React.lazy()` and `import()` for route-level chunks. This requires configuring `output.chunkFilename` and splitting entry points.
3. **Deduplicate dependencies:** Ensure there are no duplicate versions of the same package (run `bun pm ls` to inspect).
4. **Remove unused packages:** Audit `apps/mobile/package.json` for dependencies only needed by native.

### Map tiles not loading (404)

- Verify that `web-dist/basemaps/basemap.pmtiles` exists after the build.
- Check the Network tab in browser DevTools — the file should be requested at `/basemaps/basemap.pmtiles`.
- Confirm your static host supports range requests (`Accept-Ranges: bytes`). PMTiles relies on HTTP range requests for efficient tile access. Most CDNs and static hosts support this, but verify the response headers.
- If the basemap is missing from the build, the CopyWebpackPlugin pattern may not match. Run `webpack` with `--stats detailed` to see what assets were copied.

### CORS errors from the API

If the web app is hosted at `https://app.doors.example.com` and the API is at `https://api.doors.example.com`, the browser blocks cross-origin requests unless the API sends CORS headers.

**On the server side** (`apps/server/src/app.ts`), the Elysia server uses `@elysiajs/cors`. Ensure it is configured to allow your web origin:

```ts
import cors from '@elysiajs/cors'

app.use(cors({
  origin: 'https://app.doors.example.com',
  // or allow all origins during development:
  // origin: true,
}))
```

**At build time,** ensure `DOORS_API_URL` is set to the correct API origin so the client sends requests to the right host.

### SPA routing returns 404

If deployed without SPA fallback, paths like `/settings` will 404. Review section 8 and configure your host to serve `index.html` for all routes.

If the fallback is configured correctly but the page is blank, check the browser console for JavaScript errors. React Navigation's linking config may need its `prefixes` array to include the production origin — but `getWebPrefix()` already reads `window.location.origin` at runtime, so this should work automatically.

### Dev server hot reload not working

- Ensure `webpack-dev-server` ^5.2.0 is installed.
- Check for Webpack cache corruption: `rm -rf apps/mobile/node_modules/.cache`.
- Restart the dev server: `bun run dev:web`.
- If changes to `webpack.config.js` do not take effect, kill the process and restart.

### `process.env.DOORS_API_URL` is undefined

The `DefinePlugin` replaces `process.env.DOORS_API_URL` with a string literal. If the replacement fails:

- Verify the plugin entry in `webpack.config.js` matches exactly. The source code must reference `process.env.DOORS_API_URL` — not `process.env['DOORS_API_URL']` or destructured `{DOORS_API_URL}`.
- Check that the environment variable is set when running the build.

### Type errors with `@doors/api` or `@doors/server`

The Webpack aliases point directly to `.ts` source files. Babel transpiles them with the `'web'` env preset, but TypeScript type checking may fail if the packages use Node.js types not available in the browser.

If type errors block the build, run `bun run typecheck` first and resolve them. For runtime-only issues, the Babel transpilation should succeed even if TypeScript diagnostics are incomplete.
